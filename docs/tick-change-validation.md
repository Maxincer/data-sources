# Tick 变更验证记录

## 事件: INE EC (SCFIS欧线) 最小变动价位调整

- **生效日期**: 2026-05-11
- **旧 tick**: 0.1 点
- **新 tick**: 0.5 点
- **规则来源**: 上海国际能源交易中心通知（经中信期货转发）
  - 转发链接: https://www.citicsf.com/e-futures/content/000204/822721
  - ⚠️ 此为第三方转发页，非交易所原始链接，如失效需在 INE 官网查找原始公告

### 交易所规定的调整规则

2026-05-11 起，前结算价需按以下规则从旧 tick (0.1) 调整到新 tick (0.5):

| 末位数 | 调整方式 |
|--------|---------|
| 0, 5   | 不变 |
| 1, 2   | 调整为 0 |
| 3, 4, 6, 7 | 调整为 5 |
| 8, 9   | 调整为 0 并向前进 1 位 |

然后**用调整后的前结算价** + 新 tick (0.5) 计算涨跌停价位:

```
maxup   = floor(前结算(调整后) × (1 + 涨停幅度) / new_tick) × new_tick
maxdown = floor(前结算(调整后) × (1 - 跌停幅度) / new_tick) × new_tick
```

### 发现过程

2026-05-11 的数据验证报告显示: INE EC 合约中 4 个合约的 maxup 存在差异, 3 个合约的 maxdown 存在差异.

通过逐层追溯, 发现差异原因是:

1. **原始数据校验**: 四项原始依赖（前结算价、涨跌停幅度±20%、最小变动价位0.5、FLOOR取整规则）单独看均正确
2. **全日期扫描**: 发现 2026-04-21~05-08 的数据在旧 tick=0.1 下完全符合 FLOOR, 但 05-11 新表数据不符合
3. **交易所公告溯源**: 查到 INE 在 2026-05-11 调整了最小变动价位
4. **计算规则确认**: 新表代码直接用了原始前结算价 × 1.2, 漏掉了"前结算价先调整到新 tick"这一步

### 计算验证（以 EC2605 为例）

```
Step 1: 取 2026-05-08 结算价（即 2026-05-11 的前结算价）
        1799.7 (旧 tick=0.1)

Step 2: 按交易所规则调整到新 tick=0.5
        末位 7 → 调整为 5 → 1799.5

Step 3: 用调整后的前结算价计算
        maxup   = floor(1799.5 × 1.20 / 0.5) × 0.5 = 2159.0 ✅
        maxdown = floor(1799.5 × 0.80 / 0.5) × 0.5 = 1439.5 ✅
```

### 长期解决方案

采用"跟踪交易所公告 + 交叉验证 + 自动化报告"的方式:

1. **持续交叉验证**: `verifier.py` 每日比较新表 vs 原表(Wind) 的差异
2. **报告告警**: `reporter.py` 将差异汇总到飞书/邮件
3. **人工研判**: 遇到差异时，追溯交易所官方公告确认是数据错误还是规则变更
4. **规则入库**: 确认是规则变更后，更新代码中的历史参数表（tick、涨跌停幅度等）

### 类似场景

以下参数变更也可能导致类似验证差异:

| 参数 | 来源 | 触发条件 |
|------|------|---------|
| 最小变动价位 | 交易所合约规格公告 | 交易所规则修订 |
| 涨跌停板幅度 | TradingParameters API (UPPER_VALUE) | 节假日/风险调整 |
| 保证金比例 | TradingParameters API | 节假日/风险调整 |
| 合约乘数 | 合约规格 | 极少变更 |

如验证报告出现系统性差异, 优先排查上述参数是否有变更。
